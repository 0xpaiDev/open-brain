Quick Start
SHARED

Open Brain is your AI-native personal memory system. It ingests notes, decisions, and conversations — then uses Claude to extract structured knowledge, build an entity graph, and return ranked context when you need it.

💻
CLI
Use the ob command for local shell access — ingest, search, chat directly from your terminal.
🌐
REST API
HTTP API on port 8000. Integrate with any tool, script, or service using standard JSON requests.
🤖
Discord Bot
Chat in your configured channel and memories are ingested automatically — no extra commands needed.

Authentication

All /v1/* API routes require your API key in the request header. Health endpoints are public.

HTTP Header (all API calls) 
X-API-Key: your-api-key-here
ℹ️ The CLI reads API_KEY from your .env file automatically — no manual header needed for ob commands.
💚
Health Check
SHARED

Two public endpoints let you verify the system is running without authentication.

GET /health — Liveness
Always returns 200 while the process is alive. Use for uptime monitors and load balancer checks.
GET /ready — Readiness
Returns 200 only when the database is reachable. Returns 503 if DB is down. Use this before sending traffic.
CLI 
ob health
# → status=ok database=reachable
curl 
curl http://localhost:8000/health
# → {"status": "ok"}

curl http://localhost:8000/ready
# → {"status": "ok"}  or  {"status": "error", "reason": "db_down"}

Storing Memories
USER

Memories are stored as raw text and then processed asynchronously by the worker — which uses Claude to extract type, summary, importance, entities, tasks, and decisions.

✅ Deduplication: Identical content submitted within 24 hours is detected by SHA-256 hash and skipped automatically. You'll receive status: duplicate instead of queued.

Via CLI

Basic ingest 
ob ingest "We decided to use PostgreSQL for the primary database"
With source label 
ob ingest "Meeting notes: product roadmap Q2" --source meeting
ob ingest "Bug: login form crashes on Safari" --source slack
ArgumentTypeDescription
TEXTrequiredThe memory text to store
--sourceoptionalLabel for where this came from (default: "cli"). Examples: discord, slack, meeting, notes

The command returns a raw_id (UUID) and status. Save the raw_id if you need to reference this memory later.

raw_id: 3f7a1bc2-... status: queued ← or "duplicate" if same text within 24h

Via REST API

POST /v1/memory 
curl -X POST http://localhost:8000/v1/memory \
  -H "X-API-Key: YOUR_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "text": "We decided to use PostgreSQL for the primary database",
    "source": "notes"
  }'

The API returns 202 Accepted immediately. Processing happens in the background — the worker picks it up within seconds.

{ "raw_id": "3f7a1bc2-...", "status": "queued", "supersedes_id": null }

Correcting a Memory

Use supersedes_id to mark an earlier memory as outdated. The original stays in history but is flagged is_superseded: true.

POST /v1/memory (with correction) 
curl -X POST http://localhost:8000/v1/memory \
  -H "X-API-Key: YOUR_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "text": "We switched from PostgreSQL to Supabase (managed Postgres)",
    "source": "notes",
    "supersedes_id": "3f7a1bc2-..."
  }'

Via Discord Bot

Just talk in your configured Discord channel. The bot ingests messages automatically using the discord source label — no commands needed.

Search & Retrieve
USER

Open Brain uses hybrid search — combining vector similarity (Voyage AI embeddings), keyword matching (pg_trgm), importance scoring, and recency to rank results.

CLI Search

Basic search 
ob search "database decisions"
With filters 
ob search "project roadmap" --limit 20 --type decision
ob search "auth bug" --entity "Shu"
ob search "Q1 review" --from 2026-01-01 --to 2026-03-31
FlagDefaultDescription
QUERYrequiredNatural language search query
--limit10Max results to return (up to 100)
--type—Filter by type: memory, decision, task
--entity—Filter by entity name or alias (person, project, org)
--from—Earliest date, ISO 8601 (e.g. 2026-01-01)
--to—Latest date, ISO 8601

CLI Context (for LLM input)

The context command formats results into a token-budgeted block ready to paste into Claude or any LLM.

Build LLM context 
ob context "project status" --limit 5

API Search

GET /v1/search 
curl "http://localhost:8000/v1/search?q=database+decisions&limit=10&type_filter=decision" \
  -H "X-API-Key: YOUR_KEY"
ParameterTypeDescription
qrequiredSearch query text
limitoptionalMax results (default 10, max 100)
type_filteroptionalmemory | decision | task
entity_filteroptionalEntity name or alias
date_fromoptionalISO 8601 start date
date_tooptionalISO 8601 end date
{ "results": [ { "id": "abc123...", "content": "We decided to use PostgreSQL...", "summary": "PostgreSQL chosen for primary database", "type": "decision", "importance_score": 0.87, "combined_score": 0.94 } ], "query": "database decisions" }

API Context Endpoint

Returns a pre-formatted context string with token budget information — ideal for injecting into LLM prompts.

GET /v1/search/context 
curl "http://localhost:8000/v1/search/context?q=project+status&limit=5" \
  -H "X-API-Key: YOUR_KEY"
{ "context": "[1] decision — PostgreSQL chosen...\n[2] memory — Q1 review notes...", "tokens_used": 842, "tokens_budget": 4096, "items_included": 5, "items_truncated": 0 }

Get a Specific Memory

GET /v1/memory/{memory_id} 
curl http://localhost:8000/v1/memory/abc123-... \
  -H "X-API-Key: YOUR_KEY"

Tasks
USER

Tasks are extracted from memories automatically by the worker, or created manually via the API. Each task links to a parent memory.

List Tasks

GET /v1/tasks 
# All open tasks
curl "http://localhost:8000/v1/tasks?status=open" -H "X-API-Key: YOUR_KEY"

# Tasks assigned to a specific owner
curl "http://localhost:8000/v1/tasks?status=open&owner=Shu" -H "X-API-Key: YOUR_KEY"

# Tasks due soon
curl "http://localhost:8000/v1/tasks?due_before=2026-04-01T00:00:00" -H "X-API-Key: YOUR_KEY"

Create a Task

POST /v1/tasks 
curl -X POST http://localhost:8000/v1/tasks \
  -H "X-API-Key: YOUR_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "memory_id": "abc123-...",
    "description": "Set up Supabase project",
    "owner": "Shu",
    "due_date": "2026-04-15T00:00:00",
    "status": "open"
  }'

Update Task Status

PATCH /v1/tasks/{task_id} 
curl -X PATCH http://localhost:8000/v1/tasks/task-uuid \
  -H "X-API-Key: YOUR_KEY" \
  -H "Content-Type: application/json" \
  -d '{"status": "done"}'   # open | done | cancelled

Decisions
USER

Decisions are extracted automatically when you ingest a memory that describes a choice or resolution. You can also create them manually to link structured reasoning to a memory.

List Decisions

GET /v1/decisions 
curl "http://localhost:8000/v1/decisions?limit=20" -H "X-API-Key: YOUR_KEY"

# Filter by memory
curl "http://localhost:8000/v1/decisions?memory_id=abc123-..." -H "X-API-Key: YOUR_KEY"

Create a Decision

POST /v1/decisions 
curl -X POST http://localhost:8000/v1/decisions \
  -H "X-API-Key: YOUR_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "memory_id": "abc123-...",
    "decision": "Use Supabase for managed PostgreSQL",
    "reasoning": "Reduces ops overhead, includes pgvector support out of the box",
    "alternatives": ["Self-hosted Postgres on VPS", "Neon.tech", "PlanetScale"]
  }'

Entities
USER

Entities are people, projects, organizations, and tools extracted from your memories. They form a knowledge graph — each entity has aliases so fuzzy references still resolve correctly.

List & Search Entities

GET /v1/entities 
# All entities
curl "http://localhost:8000/v1/entities" -H "X-API-Key: YOUR_KEY"

# Search by name
curl "http://localhost:8000/v1/entities?q=Shu" -H "X-API-Key: YOUR_KEY"

# Filter by type
curl "http://localhost:8000/v1/entities?type_filter=person&limit=50" -H "X-API-Key: YOUR_KEY"

Add an Alias

If the same person appears under different names, add an alias so they resolve to the same entity.

POST /v1/entities/{entity_id}/aliases 
curl -X POST http://localhost:8000/v1/entities/entity-uuid/aliases \
  -H "X-API-Key: YOUR_KEY" \
  -H "Content-Type: application/json" \
  -d '{"alias": "Shu-chan", "source": "discord"}'

Merge Duplicate Entities

If the same person/project was created as two separate entities, merge the source into the target. All aliases, memory links, and relations are moved atomically.

POST /v1/entities/merge 
curl -X POST http://localhost:8000/v1/entities/merge \
  -H "X-API-Key: YOUR_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "source_entity_id": "old-uuid-...",
    "target_entity_id": "keep-uuid-..."
  }'
⚠️ Merge is destructive — the source entity is deleted. The source name is automatically added as an alias on the target so it remains searchable.

Interactive Chat
USER

Chat with Claude grounded in your Open Brain memory. Each turn fetches relevant context and injects it into the prompt automatically.

Start a chat session 
ob chat
ob chat --model claude --topic "project decisions"
ob chat --no-ingest   # don't save this conversation to memory
FlagDefaultDescription
--modelclaudeLLM to use: claude or openai
--topic—Seed phrase for initial memory search context
--no-ingestoffSkip saving this conversation to memory at session end
1
Open Brain fetches relevant memories

On each message, it searches your memory for related context and injects the top results into Claude's prompt.

2
Claude answers with full context

Responses are grounded in what you've stored — decisions, notes, entities — not just general knowledge.

3
Conversation is saved

When you exit (quit / exit / Ctrl+C), the full conversation is ingested back into memory — unless you used --no-ingest.

Starting Services
ADMIN

Open Brain runs as three independent services: API (FastAPI), Worker (background processor), and optionally a Discord bot. They can be started locally or via Docker Compose.

Local Development (no Docker)

Terminal 1 — API server 
uvicorn src.api.main:app --reload --host 0.0.0.0 --port 8000
Terminal 2 — Worker (continuous polling) 
python -m src.pipeline.worker
Terminal 3 — Worker (process one batch & exit) 
ob worker --sync   # useful for cron jobs or one-off processing

Helper Scripts

start.sh / stop.sh 
./start.sh    # starts API + Worker + Discord bot as background processes
./stop.sh     # stops all three

# View live logs
tail -f /tmp/ob-api.log /tmp/ob-worker.log /tmp/ob-bot.log

Docker Compose (Production)

First-time setup — apply migrations 
docker compose --profile migrate up
Start services 
docker compose --profile api up -d       # FastAPI server
docker compose --profile worker up -d    # Background worker
docker compose --profile discord up -d   # Discord bot (optional)
docker compose --profile caddy up -d    # TLS reverse proxy (optional)
Check status 
docker compose ps
docker compose logs -f api --tail=50
docker compose logs -f worker --tail=50
ℹ️ Each service uses json-file logging with 50MB max size and 5-file rotation. Docker handles log rotation automatically.

Worker & Queue Health
ADMIN

The worker processes memories from the refinement_queue table using SELECT FOR UPDATE SKIP LOCKED — safe for multiple workers. Failed jobs land in dead_letters after 3 attempts.

Queue Status

GET /v1/queue/status 
curl http://localhost:8000/v1/queue/status -H "X-API-Key: YOUR_KEY"
{ "pending": 3, ← waiting to be picked up "processing": 1, ← currently being processed "done": 142, ← successfully processed "failed": 2, ← moved to dead letters "total": 148, "oldest_locked_at": "2026-03-18T03:12:00" ← worker health signal }
⚠️ Stale locks: If oldest_locked_at is more than 5 minutes old (WORKER_LOCK_TTL_SECONDS default: 300), the worker may be stuck. Restart the worker service. Stale jobs are automatically reclaimed on next poll.

Dead Letters

Jobs that fail 3 times move to the dead letter queue. Inspect and retry them here.

View dead letters 
# Unresolved failures
curl http://localhost:8000/v1/dead-letters -H "X-API-Key: YOUR_KEY"

# Resolved (previously retried) failures
curl "http://localhost:8000/v1/dead-letters?resolved=true" -H "X-API-Key: YOUR_KEY"
Retry a failed job 
curl -X POST http://localhost:8000/v1/dead-letters/failed-uuid/retry \
  -H "X-API-Key: YOUR_KEY"
{ "id": "failed-uuid", "queue_id": "queue-uuid", "retry_count": 1, "message": "Re-enqueued for processing" }
ℹ️ The retry limit is controlled by DEAD_LETTER_RETRY_LIMIT. A 409 response means the limit has been reached — investigate error_reason and last_output in the dead letter record before retrying manually.

Daily Aggregations
ADMIN

The importance job runs daily and updates dynamic_importance on memory items based on how often they've been retrieved (retrieval_events table). The computed column importance_score recalculates automatically.

Run Manually

Direct Python 
python -m src.jobs.importance
Via Docker 
docker compose run --rm worker python -m src.jobs.importance

Automated Cron Setup

crontab -e 
# Daily at 3:00 AM — importance score aggregation
0 3 * * * cd /opt/open-brain && \
  docker compose run --rm worker python -m src.jobs.importance \
  >> /var/log/openbrain-jobs.log 2>&1
ℹ️ Do not update importance_score directly. It is a PostgreSQL GENERATED column. Update base_importance or dynamic_importance only — the score recalculates automatically.
🔮
Synthesis Job
ADMIN

The synthesis job produces a structured weekly digest — grouping memories by entity, identifying patterns, and storing the result as a new memory item with source: synthesis. It uses Claude to generate the digest.

Trigger via API

POST /v1/synthesis/run 
curl -X POST http://localhost:8000/v1/synthesis/run \
  -H "X-API-Key: YOUR_KEY" \
  -H "Content-Type: application/json" \
  -d '{"days": 7}'   # 1–90 days lookback
{ "synthesis_id": "synth-uuid-...", "memory_count": 47, "date_from": "2026-03-11", "date_to": "2026-03-18", "skipped": false, "message": "Synthesis complete. 47 memories clustered into 8 topics." }

Run Manually

Direct Python 
python -m src.jobs.synthesis

Automated Cron Setup

crontab -e (weekly Sunday 2 AM) 
# Weekly synthesis — Sunday at 2:00 AM
0 2 * * 0 cd /opt/open-brain && \
  docker compose run --rm worker python -m src.jobs.synthesis \
  >> /var/log/openbrain-jobs.log 2>&1
⚠️ Model note: Synthesis currently uses claude-haiku-4-5 for cost efficiency. Set SYNTHESIS_MODEL=claude-opus-4-6 in .env before going to production for best quality digests.

Database Administration
ADMIN

Migrations

Apply all pending migrations 
alembic upgrade head

# Or via Docker
docker compose --profile migrate up
Check migration state 
alembic current    # which revision is applied
alembic history    # full migration history
alembic heads      # latest revision
🚫 Never use Base.metadata.create_all() against Supabase. Always use alembic upgrade head. SQLAlchemy creates the embedding column as JSONB; only Alembic correctly sets it to vector(1024).

Backup

Manual pg_dump 
pg_dump "$SQLALCHEMY_URL" > backup_$(date +%Y%m%d).sql
Automated backup cron (daily 3:30 AM, 30-day retention) 
30 3 * * * cd /opt/open-brain && \
  ./scripts/backup.sh >> /var/log/openbrain-backup.log 2>&1
⚠️ Use the Supabase direct connection (port 5432) — not PgBouncer (port 6543). SELECT FOR UPDATE SKIP LOCKED breaks on the pooler connection.
