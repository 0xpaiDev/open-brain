Engineering · Design Decision

Why not LangChain or Swarm?

🦜 LangChain

  • Abstraction leaks — you still need to understand the underlying API calls to debug failures
  • Version churn — breaking changes every minor release
  • Callback hell for even simple retry logic
  • Opaque chain tracing — hard to know why the LLM got bad input
  • Built for chains; adds overhead for a single extraction step

🐝 OpenAI Swarm

  • Designed for multi-agent handoffs — we have exactly one worker doing one job
  • Context variables passed between agents don't map to our PostgreSQL-centric model
  • Agent loop overhead for a linear pipeline is pure waste
  • Handoff semantics lose the durable queue guarantees we need for crash recovery
  • Experimental — not production-ready at time of build

✓ Queue-Worker Pattern

  • Durable by default — jobs survive worker crashes via stale lock reclaim
  • Observable — every stage is a plain async Python function, easy to log and trace
  • Zero framework tax — Tenacity for retry, structlog for tracing, that's it
  • PostgreSQL guarantees ordering, idempotency, and consistency
  • Scale when needed: add workers, don't change architecture

The right orchestration for this system is PostgreSQL's SELECT FOR UPDATE SKIP LOCKED — not an agent framework.

Engineering · State Management

PostgreSQL is the state machine

Immutability Rules

Append-Only raw_memory
Never UPDATE or DELETE. Corrections create new memory_items with supersedes_id. Full audit trail by design.
Composite PK Idempotency
(memory_id, entity_id) on junction tables + ON CONFLICT DO NOTHING = reprocessing the same job twice is always safe.
Stale Lock Reclaim
Worker claims jobs with locked_at < now() - 5min. Crashed workers release automatically. No manual intervention.

GENERATED Column (single source of truth)

-- Never UPDATE importance_score directly
importance_score FLOAT
  GENERATED ALWAYS AS (
    0.6 * base_importance
    + 0.4 * dynamic_importance
  ) STORED

-- dynamic_importance updated daily by cron
-- decays exponentially via retrieval events
No Redis No in-process cache No shared mutable state across requests
Engineering · Worker Pipeline

6 stages, each independently failable

📐 Normalize NFC, collapse blanks, tokenize, 2k-token chunks
→
🤖 Extract Claude Haiku → JSON entities, decisions, tasks, importance
→
✅ Validate Pydantic schema, entity name normalization
→
🧬 Embed Voyage AI voyage-3 1024-dim vectors, Tenacity 3×
→
🔗 Resolve pg_trgm fuzzy match ≥0.92, merge aliases, exact→fuzzy→insert
→
💾 Store Transactional: memory_items + entities + links + commit()
ExtractionFailed
Attempts < 3 → reset to pending, escalate prompt. Attempts ≥ 3 → dead letter (failed_refinements). Inspectable + retryable via API.
EmbeddingFailed
Not retryable with escalation. Tenacity handles rate limits (3× backoff 2/4/8s). After exhaustion → dead letter immediately.
Engineering · Retrieval

Hybrid Search Formula

combined_score =
  0.50 × vector_similarity  // semantic meaning
  + 0.20 × keyword_relevance  // exact term matching
  + 0.20 × importance_score  // base×0.6 + dynamic×0.4
  + 0.10 × recency_score  // exp(-ln2/30 × days)
HNSW Index (vector search)
Cosine distance on 1024-dim Voyage embeddings. Top 100 candidates, then re-ranked. Fast approximate nearest-neighbour via pgvector HNSW index.
GIN Index (full-text search)
to_tsvector('english', content) — the FTS query expression must be identical to the index definition or the GIN is bypassed (learned the hard way).
Importance Score
GENERATED column. dynamic_importance updated nightly by exponential-decay aggregation of retrieval_events. Frequently-accessed memories rise.
Recency Score
Exponential decay with 30-day half-life. A memory accessed today scores 1.0; same memory after 30 days scores 0.5. All weights are env-var configurable.
Engineering · Resilience

Built to never lose a job

1
SELECT FOR UPDATE SKIP LOCKED
Distributed-safe job claiming. Multiple workers can run simultaneously — each gets a different job. No external queue needed at MVP scale.
2
Stale Lock Reclaim (locked_at < now() - 5min)
Worker crashed? Jobs stuck in processing are automatically reclaimed after 5 minutes. No manual intervention. No lost work.
3
Composite PKs + ON CONFLICT DO NOTHING
Processing the same job twice = safe. All junction tables use composite PKs. All inserts use conflict-ignore. The worker is fully idempotent.
4
Content-Hash Dedup (24-hour window)
SHA-256 of normalized content prevents duplicate ingestion storms — especially from Discord where a bot might re-send the same message multiple times.
5
3-Attempt Dead Letter Queue
Extraction failures retry with escalating prompts (attempts 1→2). On attempt 3, the job moves to failed_refinements — inspectable and retryable via POST /v1/dead-letters/{id}/retry.
Engineering · Architecture Deep Dive

The honest answer about orchestration

What Swarm solves (and we don't need)

# Swarm is great for:
agent_a = Agent("extractor")
agent_b = Agent("embedder")
# → handoff between agents
# → context variables flow across
# → each agent can call tools

# We have: one worker, one LLM call,
# one pipeline. Swarm overhead = zero value.
The real "agent handoff" here
Status transition in refinement_queue: pending → processing → done. PostgreSQL is the orchestrator. Each stage reads from and writes to the DB. Crash-safe, inspectable, no framework needed.

Our actual pipeline (transparent Python)

async def process_job(queue_row, ...):
  # Each step = plain async function
  chunks  = await normalize(raw.text)
  extract = await extract_entities(chunks)
  # ExtractionFailed → retry or dead letter
  embed   = await embed_text(chunks)
  # EmbeddingFailed → dead letter immediately
  entities = await resolve_entities(session, ...)
  await store_memory_item(session, ...)
  await session.commit()  # REQUIRED (learned)
Full stack trace on failure No framework version to chase Structlog spans the entire job

Scheduled jobs, not async loops

📝 Weekly Synthesis (2 AM Sunday)

Fetches last 7 days of memories + entity names → sends to Claude for theme clustering → stores result as a new memory_item with type='context', base_importance=0.8.

# Defaults to Haiku; set Opus for production
SYNTHESIS_MODEL=claude-opus-4-6
python -m src.jobs.synthesis

🤔 Why External Cron, Not Async Loop?

Long-running Claude Opus calls (synthesis can take 20–30s) must not block the async event loop. External cron = process isolation, crash containment, no impact on API latency.

🔌 Why Not APScheduler / Celery Beat?

These run inside the Python process — which defeats the point. Host cron spawns an independent process. Simple, auditable, doesn't require another service to keep running.

Engineering · Hard-Won Lessons

5 gotchas that cost real debugging time

1
session.flush() ≠ session.commit() — the silent data loss
Worker was logging process_job_success but DB stayed empty. SQLAlchemy's async context manager calls session.close() on exit — implicit rollback. You must await session.commit() explicitly after every terminal operation.
2
pgvector codec + register_vector() = TypeError in production
pgvector.sqlalchemy.Vector.process_bind_param returns a string like '[0.1,...]'. The asyncpg register_vector codec expects a list. Do NOT call register_vector — let PostgreSQL cast text → vector natively.
3
GIN index not used — FTS expression must be byte-for-byte identical
The Alembic migration creates the index on to_tsvector('english', content). Your search query must use the same expression — same function, same config string. Any variation forces a full table scan.
4
Settings singleton captures .env at import time — breaks test isolation
When a .env file exists, config.settings loads production keys at module import. Use an autouse fixture: monkeypatch.setattr(_config, "settings", _config.Settings()) to re-create from test env vars each test.
5
ORM identity map conflict after raw SQL UPDATE + session.delete()
After raw SQL updates on child tables, calling session.delete(entity) makes SQLAlchemy try to SET NULL stale FK references — which crashes on composite PK columns. Fix: session.expunge(entity) first, then delete via SQLAlchemy Core.
