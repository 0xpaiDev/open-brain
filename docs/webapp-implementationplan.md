# Open Brain Daily Dashboard — Implementation Plan

## Context

Open Brain is a personal memory/knowledge system with a FastAPI backend, PostgreSQL + pgvector, and a Discord bot interface. The morning pulse (daily check-in), task management, and memory ingestion all exist as backend features but are only accessible via API calls or Discord. This plan adds a Next.js web dashboard that unifies the daily workflow: morning pulse, calendar, tasks, and memory ingestion in one place.

---

## 1. Audit Summary

### Morning Pulse
- **Model**: `DailyPulse` (table `daily_pulse`) — id, pulse_date (UNIQUE), raw_reply, sleep_quality (int 1-5), energy_level (int 1-5), wake_time (String(10) HH:MM), parsed_data (JSONB), ai_question, ai_question_response, notes, status ("sent"/"replied"/"parsed"/"completed"/"skipped"/"expired"), discord_message_id, created_at, updated_at
- **API**: `POST /v1/pulse` (create), `GET /v1/pulse/today` (fetch or 404), `PATCH /v1/pulse/today` (update fields via `PulseUpdate` model)
- **Question of the Day**: LLM-generated by Claude Haiku. Alternates operational vs reflective. Uses open todos as context. Fallback: "What's one thing you want to accomplish today?"
- **Cron**: 05:00 UTC daily via supercronic. Creates DailyPulse record (status="sent"), generates AI question, fetches calendar + todos, sends Discord DM.

### Calendar Events
- **NOT persisted in DB.** Fetched live from Google Calendar during cron job, used to build Discord embed, then discarded.
- **Model**: `CalendarState` (Pydantic) — `events: list[CalendarEvent]` (title, start, end, location, calendar, all_day), `tomorrow_preview: list[CalendarTomorrowEvent]` (title, start, all_day), `date`, `fetched_at`
- **Function**: `fetch_today_events(settings)` in `src/integrations/calendar.py` — gracefully degrades to empty state if Google libs missing or credentials expired.

### Task System (Two Systems)
1. **Task** (table `tasks`) — extracted from memory pipeline. Columns: id, memory_id (FK), description, owner, due_date, status ("open"/"done"/"cancelled"), created_at. API: `GET/POST/PATCH /v1/tasks`
2. **TodoItem** (table `todo_items`) — primary system, Discord-managed. Columns: id, description, priority ("high"/"normal"/"low"), status ("open"/"done"/"cancelled"), due_date, discord_message_id, discord_channel_id, created_at, updated_at. API: `POST /v1/todos`, `GET /v1/todos` (filters: status, priority, due_before, limit, offset), `GET /v1/todos/{id}`, `PATCH /v1/todos/{id}`, `GET /v1/todos/{id}/history`
3. **TodoHistory** (table `todo_history`) — audit log. Columns: id, todo_id (FK), event_type, old_value (JSONB), new_value (JSONB), reason, created_at
4. **No `source` field** on TodoItem — no calendar vs manual distinction exists

### Memory Ingestion
- **Endpoint**: `POST /v1/memory` (202 Accepted) — body: `{text (1-50000 chars), source (default "api"), metadata (JSONB, max 8KB), supersedes_id}`
- **Response**: `{raw_id, status ("queued"/"duplicate"), supersedes_id}`
- **Dedup**: SHA-256 hash within 24h window
- **Single-item fetch**: `GET /v1/memory/{memory_id}` — returns `MemoryItemResponse` (id, raw_id, type, content, summary, base_importance, dynamic_importance, importance_score, is_superseded, supersedes_id, created_at)
- **No list endpoint** — cannot browse memories without searching

### Auth
- **X-API-Key header only** on all `/v1/*` routes. No sessions, JWT, or cookies.
- **CORS**: `allow_origins=[]` — browser requests are currently blocked.
- **Public paths**: `/health`, `/ready` only
- **Middleware**: `APIKeyMiddleware` in `src/api/middleware/auth.py`

### New Endpoint: `GET /v1/calendar/today`

**File**: `src/api/routes/calendar_api.py` (new)

Thin wrapper around `fetch_today_events(settings)` from `src/integrations/calendar.py`.

```
Response 200:
{
  "status": "ok",
  "date": "2026-04-02",
  "fetched_at": "2026-04-02T08:30:00Z",
  "events": [
    {"title": "Team standup", "start": "2026-04-02T09:00:00+03:00", "end": "...", "location": null, "calendar": "Work", "all_day": false}
  ],
  "tomorrow_preview": [
    {"title": "Dentist", "start": "2026-04-03T10:00:00+03:00", "all_day": false}
  ]
}
```

`status` field: `"ok"` when Google Calendar credentials are valid and events were fetched, `"unavailable"` when credentials are missing/expired/libs not installed. This lets the frontend distinguish "no events today" from "calendar not connected" — `fetch_today_events()` returns empty `CalendarState` in both cases (by design, it never raises).

Register in `main.py`. Rate limit: 60/min.

### New Endpoint: `GET /v1/memory/recent`

**File**: `src/api/routes/memory.py` (add to existing)

```
GET /v1/memory/recent?limit=20&offset=0&type_filter=memory

Response 200:
{
  "items": [MemoryItemResponse, ...],
  "total": 142
}
```

Query: `SELECT * FROM memory_items WHERE is_superseded = false [AND type = :type_filter] ORDER BY created_at DESC LIMIT :limit OFFSET :offset`

Reuses existing `MemoryItemResponse` model.

### Auth Strategy

**API key via localStorage** (appropriate for single-user personal tool):

**`lib/api.ts`**:
```typescript
const API_KEY_STORAGE_KEY = 'ob_api_key'

export async function api<T>(method: string, path: string, body?: unknown): Promise<T> {
  const key = localStorage.getItem(API_KEY_STORAGE_KEY)
  if (!key) throw new Error('NOT_AUTHENTICATED')
  
  const res = await fetch(path, {
    method,
    headers: { 'X-API-Key': key, 'Content-Type': 'application/json' },
    body: body ? JSON.stringify(body) : undefined,
  })
  
  if (res.status === 401) {
    localStorage.removeItem(API_KEY_STORAGE_KEY)
    throw new Error('NOT_AUTHENTICATED')
  }
  if (!res.ok) throw new Error(`API error: ${res.status}`)
  return res.json()
}
```

**Auth gate**: `app/layout.tsx` wraps children in an `<AuthProvider>` context. If no key in localStorage, render a full-screen auth dialog (shadcn `<Dialog>` with `<Input>` for API key). On submit, test with `GET /v1/pulse/today`. If 200 or 404 → key is valid, store and proceed. If 401 → show error.

### State Management

React state + Context. No external library for MVP.

- **`AuthContext`**: API key, isAuthenticated, logout()
- **Custom hooks** (`use-pulse`, `use-todos`, `use-calendar`, `use-memories`): each hook encapsulates fetch + state for its domain. Uses `useEffect` for initial fetch, returns `{ data, loading, error, mutate }`.
- **No global store**: Each page manages its own data via hooks. No cross-page state needed for MVP.

## 6. Open Questions

1. **Manual pulse creation**: If the user opens the dashboard before the 05:00 UTC cron, should "Start your day" create a pulse with the default question, or should it call the LLM to generate one? **Recommendation**: default question (no LLM call from web — keep it simple).

2. **Calendar caching**: `GET /v1/calendar/today` hits Google Calendar API on every request. Add a simple in-memory TTL cache (5 min) in the endpoint? **Recommendation**: yes, module-level `_cache` dict with timestamp check.

3. **Todo sort order**: `GET /v1/todos` has no server-side ORDER BY. **This is a bug** — without ORDER BY, pagination with `offset`/`limit` is non-deterministic (rows can appear in multiple pages or be skipped after VACUUM). Particularly bad for `status=done&limit=10` which returns arbitrary completed todos instead of the most recent. **Resolution**: Add `.order_by(TodoItem.created_at.desc())` to `list_todos()` in `src/api/routes/todos.py` before `.offset().limit()` as a prerequisite backend fix.

4. **Search scope**: TopNav search on Memory tab only or everywhere? **Recommendation**: Memory tab only for MVP.

5. **File upload (Media tab)**: Defer to Phase 2 or implement now? **Recommendation**: Phase 2. Show dropzone UI with "Coming soon" badge.

6. **Docker deployment**: Next.js needs its own Dockerfile in `web/` and a new service in `docker-compose.yml`. Caddy config needs updating to route `/*` → Next.js and `/v1/*` → API. **Recommendation**: Add `web` service to docker-compose with `profiles: [web]` so it's opt-in like the scheduler.
